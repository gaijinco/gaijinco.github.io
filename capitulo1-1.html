¡Hola Mundo!

Las funciones son un concepto fundacional en la programación. T an importantes son que vale la pena que sean lo primero que aprendamos.

Una función toma unos parámetros o input y produce un valor de retorno o output. Los pasos que se siguen para esa transformación reciben el nombre de algoritmo.

Ejemplos sencillos de una función son:

  • Un procedimiento que calcule el precio que el consumidor debe pagar, teniendo como input el precio base y el valor del impuesto a cobrar, y como output el precio final.
  • Un procedimiento que devuelva sinónimos de una palabra, teniendo como input la palabra que nos interesa, y como output una lista de palabras.
  • Un procedimiento que indique si un año es bisiesto, teniendo como input el número del año y como output un predicado verdadero si es bisiesto o falso en caso contrario.
  • Una función que resuelva una sopa de letras, teniendo como input una cuadrícula cuadrada con letras mayúsculas y una lista de palabras a buscar, y como output una sopa de letras donde las palabras de la lista aparezcan destacadas.

Los algoritmos son secuenciales, llevamos a cabo un primer paso antes de llevar a cabo un segundo paso, etc. hasta que finalmente somos capaces de devolver el output esperado.

Revisemos cada concepto en más detalle.

Valor de Retorno
El valor de retorno del código 01 es String, que representa una cadena de caracteres. Sin embargo las funciones pueden retornar otros tipos de datos, como números, fechas o incluso tipos de datos que nosotros mismos definamos.
Identificadores
El identificador del código 01 es <saludos>. El uso de un buen identificador es muy importante para nuestra capacidad de entender el código que nosotros escribimos o que escriben otras personas. Sin embargo no todos los identificadores son válidos, hay unas reglas básicas. Aquí las más importantes:
Identificador ¿Válido? Explicación
mi saludo maravilloso ❌ Los identificadores no pueden tener espacios.
101saludos ❌ Los identificadores pueden tener números, pero no pueden empezar con ellos.
SopaDeLetras ✅ Los identificadores pueden llevar mayúsculas y minúsculas.
CONSTANTE_GRAVITACIONAL ✅ Los identificadores pueden usar "_", por ejemplo para separar palabras en mayúsculas.

Si usas un identificador que no sea válido, tu programa no va a compilar y a veces tendrá un mensaje un poco críptico. Afortunadamente hay una manera sencilla de evitar estos errores: evita los espacios y los números como caracteres iniciales, lo demás es un poco intuitivo.

Las mayúsculas y minúsculas son importantes en los identificadores que uses. Una función que se llama saludos es diferente de una que se llame Saludos. Si no usas el identificador exacto de una función obtendrás un error de compilación.

Hay algunas convenciones en el mundo de la programación que conviene seguir. La más usada en lenguajes como Java es el uso de Camel Case: Escribe primordialmente en minúsculas pero si un identificador usa más de una palabra entonces haz que la letra inicial esté en mayúsculas. Algunos ejemplos serían: SopaDeLetras, miSaludo, unIdentificadorMuyLargo, etc. Si además la primera letra es también mayúscula se llama Uppercase Camel Case, sino se llama Lowercase Camel Case. En Java la convención es que las funciones y las variables utilicen Lowercase Camel Case.

Todas las funciones deben terminar en una sentencia return, es decir una sentencia que inicia con la palabra clave return. Una sentencia return indica el valor que se va a retornar una vez que la función termina de ser ejecutada. En la función <saludos> siempre retornamos la misma cadena de caracteres: "¡Hola Mundo!"

Pronto escribiremos funciones más complejas. Las funciones pueden tener múltiples sentencias, cada una haciendo un paso intermedio para calcular el valor final. Lo más importante a tener en cuenta es que una función se ejecuta una línea a la vez, en el orden que aparecen.

Uno de los errores más comunes al aprender a programar en Java es olvidar poner el ";" al final de una sentencia. ¡No olvides tus ";"!

Los ";" son importantes porque permiten al compilador identificar dóndetermina una sentencia, de igual manera que los humanos necesitamos un "." para saber dónde termina una oración. Hay otros lenguajes que usan el caracter de final de línea para indicar el final de una sentencia. Por lo que no se puede poner más una sentencia por línea de código, algo que antes los programadores veían como inconveniente aunque hoy en día es mucho más aceptado. Por lo tanto los lenguajes deben tomar una decisión: permitir más de una sentencia por línea, necesitando un caracter como ";" para separarlas, o permitir una sola sentencia por línea y no usar ningún delimitador.

DRY es una sigla para Don't Repeat your Code. Si tenemos repetición en nuestro código, cuando queramos ampliarlo o corregirlo tenemos que tener cuidado de hacer los cambios en todas las partes. Si olvidamos hacerlo entonces tendremos un sistema inconsistente y a veces eso genera errores sutiles difíciles de encontrar. Al evitar la repetición nos aseguramos de evitar este tipo de problemas.

El Código 03 también nos reitera que dentro de una función podemos usar funciones dentro de funciones incluso las que hemos definido nosotros. No solo es permitido sino que es algo que debemos fomentar. En un sentido, la programación trata de hacer pequeñas partes y después usar esas pequeñas para crear partes más complejas.

Es muy importante que empieces a ejercitar el hábito de escribir pequeñas funciones y utilizarlas como partes de otra función. Lo inverso es también importante, si vas a crear una función, piensa cómo puedes dividir el algoritmo en partes más pequeñas y escribir una función para cada una.

Cuando estés escribiendo una función, piensa en cómo puedes implementarla en términos de funciones más pequeñas. Esto te dará la oportunidad de reutilizarlas más adelante además hará que tus funciones sean más fáciles de leer y de probar.

Espacio en blanco

El espacio en blanco es cualquier caracter que parezca un espacio, esto incluye espacios, tabulaciones y saltos en línea. El espacio en blanco no tiene ninguna afectación en el funcionamiento de una función, es netamente una opción para mejorar la legibilidad para los seres humanos. Cuánto espacio en blanco es el exacto para mejorar la legibilidad de un código es una cuestión subjetiva, pero parece claro que el Código 04 tiene mucho y el Código 05 tiene muy poco. Por lo tanto, busca escribir código con una cantidad de espacio en blanco que esté entre esos dos estilos, pero sobre todo no temas agregar un poco de espacio en blanco para darle espacio a respirar a los elementos que componen una sentencia para hacerlos más fáciles de entender.

Después de escribir una función, dale un vistazo y mira si se siente densa. En caso de duda, pide una segunda opinión. Agrega un poco de espacio en blanco para hacerla más sencilla a los ojos.

Comentarios

La línea 06 tiene un comentario de una sola línea, empieza con “//” y termina en el siguiente salto de línea. Los comentarios son ignorados por el compilador, son realmente para dejar notas para las personas que puedan leer el código.

Dejar notas en tu código puede ayudar a que sea más fácil de entender. Pero ten cuidado de que el comentario no se desactualice si cambias el código.

También existen los comentarios de múltiples líneas, que empiezan con /* y terminan con */ y como su nombre lo indica pueden englobar múltiples líneas. No es necesario, pero tiende a usarse, que cada línea intermedia empiece con un * para darle un aspecto estético. Este es un comentario de múltiples líneas:

/*
 * Este es un comentario de múltiples líneas
 */

Los comentarios de una sola línea son usualmente preferidos porque es muy sencillo comentar mucho código, y después descomentar partes de él. Sin embargo es importante conocerlos porque los vas a encontrar en muchas partes.
